Write-ahead logging ensures durability by persisting changes to a sequential log before applying them to the main data structure. The log is an append-only sequence of entries describing intended mutations; because sequential writes are significantly faster than random writes on spinning disks and nearly equivalent on SSDs, WAL is a fundamental durability primitive. Recovery after a crash replays committed log entries and discards uncommitted ones. Log-structured merge trees extend this pattern by keeping all writes sequential and periodically compacting sorted runs. PostgreSQL uses WAL for both crash recovery and replication: followers replay the primary's WAL stream to maintain synchronized state. The major operational concern with WAL is log bloat when vacuum or compaction cannot keep pace with write volume.
