Event sourcing stores the history of state changes as an immutable sequence of events rather than the current state. The current state is derived by replaying the event log from the beginning or from a snapshot. This provides a complete audit trail, enables temporal queries against past states, and simplifies debugging by showing exactly what happened and when. The write model appends events; the read model is maintained by projecting events through view-building consumers. The event schema must be versioned carefully because schema changes affect replay of historical events. Event sourcing pairs naturally with CQRS because writes (appending events) and reads (querying projections) have completely different performance characteristics. Kafka is commonly used as the event log for systems that need horizontal scalability. The primary scaling challenge is replay time for long-lived aggregates with large event histories.
