Hot shard detection identifies database partitions receiving disproportionate traffic, typically defined as any shard receiving more than twice the average request rate. The root cause is usually non-uniform key distribution, where certain keys are accessed far more frequently than others. Detection approaches include monitoring shard-level QPS in the storage layer and surfacing statistical outliers. Mitigation strategies include splitting hot shards into smaller partitions, adding read replicas for read-heavy hot spots, caching frequently accessed values above the storage layer, and routing writes through a scatter-gather pattern that shards a single hot key across multiple partitions. DynamoDB automatically splits hot partitions when they exceed provisioned capacity thresholds. TiDB continuously monitors and rebalances regions based on access frequency using a placement driver that tracks per-region statistics.
