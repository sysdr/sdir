Quorum systems coordinate reads and writes across replicas using configurable thresholds. In a replicated datastore with N replicas, write quorum W and read quorum R satisfy R + W > N to ensure every read overlaps at least one successful write, providing strong consistency under normal operation. If R + W <= N, stale reads become possible because a read may contact replicas that never observed the latest write. Quorum choices trade latency for consistency: low W improves write latency, low R improves read latency, and high values improve freshness and fault tolerance. Dynamo-style databases expose tunable consistency with options like ONE, QUORUM, and ALL to let applications choose this trade-off per request.
